# TICKET-IMP-08 — Create a new account directly from the CSV import step

- **Area:** CSV Import
- **Traceability:** extends FR-IMP-1 (needs an existing account before rows commit)

## User story

As a user importing a CSV for an account that isn't in the app yet — including a first-ever import with no accounts set up at all — I want a clear, prominent way to create that account right from the import step instead of cancelling the import to go create it first, so a first-time import of a new bank/account doesn't require leaving the wizard, and doesn't leave me stuck staring at an empty dropdown.

## Description

Step 1 of the import wizard (account selection) currently requires every queued file to be linked to an **existing** account — there's no way to create one inline, and if the user has **no accounts yet** (true first run), step 1 is a dead end with nothing to pick. This adds a standalone **"+ New account" quick action** in step 1 — not buried as one more `<select>` option — that pre-fills a new account from the file's own detected IBAN (or the filename), nudges/defaults toward itself when no accounts exist yet, and lets a batch of files destined for the same not-yet-created account all link to a single pending creation instead of each spawning its own account.

## Current situation (as-is)

- [import-select-step.component.html:42-51](../../../src/app/feature-import/components/import-select-step/import-select-step.component.html) renders a native `<select>` per queued file, bound to `row.accountId`, populated only from `accounts()` (existing `Account[]` passed in from `accountsStore.activeAccounts()`, see [import-wizard.component.html:62](../../../src/app/feature-import/components/import-wizard/import-wizard.component.html)). There is no "create new account" entry.
- `QueuedImportFile` ([import-select-step.component.ts:14-18](../../../src/app/feature-import/components/import-select-step/import-select-step.component.ts)) is `{ file, accountId: number | null, autoDetected: boolean }` — `accountId` can only reference an already-persisted `Account.id`.
- [`canAdvanceFromStep1`](../../../src/app/feature-import/components/import-wizard/import-wizard.component.ts) (`import-wizard.component.ts:100-102`) requires `queue().every(row => row.accountId !== null)`, so today the only way past step 1 for a new account is to cancel, create the account via [account-form.component.ts](../../../src/app/feature-accounts/components/account-form/account-form.component.ts), and re-open the import wizard.
- **No empty-accounts handling at all.** When `accounts()` is empty (a genuinely first-time user, no accounts created yet), the `<select>` at [import-select-step.component.html:42-51](../../../src/app/feature-import/components/import-select-step/import-select-step.component.html) still renders — just the disabled `"Choose an account"` placeholder with zero real options underneath it (the `@for (account of accounts(); ...)` loop yields nothing). The user can drop files in, see them queued, and then has no way to advance: no account to pick, no visible next step, no hint that account creation is even possible from here.
- Own-account IBAN **is** already detected per file today: [`detectOwnIban`](../../../src/app/core/import/account-detection.ts) (`account-detection.ts:9-27`) reads the mapping profile's `columns.ownIban` header and returns the first non-empty sample value; [`matchAccountByIban`](../../../src/app/core/import/account-detection.ts) (`account-detection.ts:30-37`) then matches it against existing accounts by normalized IBAN. This runs per file in `detectAccountId` (`import-select-step.component.ts:96-121`), called from `addFiles` — but today a detected IBAN that matches **no** existing account is simply discarded (`accountId` stays `null`, `autoDetected` stays `false`); nothing offers to create an account from it.
- `Account` ([app-db.ts](../../../src/app/core/data-access/app-db.ts)) has `name: string`, `type: 'checking' | 'savings' | 'joint' | 'invest'`, `iban?: string`, plus `currency`, `openingBalance`, `openingBalanceDate`, `color`, `icon`, `archived`. [`AccountsStore.addAccount`](../../../src/app/core/state/accounts.store.ts) (`accounts.store.ts:188`) persists through [`AccountsRepository.add`](../../../src/app/core/data-access/accounts.repository.ts) — components never write `appDb.accounts` directly.
- There is no existing pattern letting one queued file's step-1 selection be referenced by another queued file — the only cross-file linkage today is batch column-mapping ([TICKET-IMP-02](../../v1.0_foundation/tickets/TICKET-IMP-02-batch-multi-file-mapping.md)), which shares a `MappingProfile`, not an account.

## Desired result (to-be)

- Each queued file row gets a **"+ New account" quick action** — a visible button next to the account `<select>`, not an option hidden inside the dropdown's option list — so the ability to create an account is obvious at a glance rather than something the user has to discover by opening the dropdown.
- **Empty-accounts nudge:** when `accounts()` is empty (no accounts exist in the app at all), step 1 doesn't just render a dead-end disabled dropdown. It actively steers the user toward account creation — e.g. the dropdown is replaced/fronted by the "+ New account" quick action in a prominent/default state (auto-opened or visually emphasized) for every queued file, with copy along the lines of "No accounts yet — create one to import into," so a genuine first-time user isn't stuck guessing what to do next.
- Triggering the quick action reveals an inline, editable draft for that file: `name` and `iban` pre-filled from `detectOwnIban`'s result for that file if one was found; otherwise `name` pre-fills from the file's filename (extension stripped) and `iban` is left blank. `type` defaults to `'checking'`. The user can edit any of these before continuing.
- Only **one** file per new account actually creates it. When a second (or third, ...) queued file is meant to link to the **same** not-yet-created account, that file's dropdown instead offers the pending draft as a selectable option, labeled with the owning file's filename (e.g. `"→ New account (from rabobank-jan.csv)"`) — picking it links that file to the same to-be-created account without opening a second draft or duplicating account creation.
- The account is actually persisted (via `AccountsStore.addAccount`, never a raw `appDb.accounts.add`) once — at the point the owning file's rows are committed (consistent with `runCommit()`'s existing per-file commit flow) so a user backing out of the wizard before committing doesn't leave an orphan account behind. All files linked to that draft resolve to the same real `Account.id` once it's created.
- `canAdvanceFromStep1` treats a file with a valid, non-empty draft (or a link to one) as satisfied, the same as a file with a persisted `accountId`.
- Existing single-account-per-file behavior (existing accounts, IBAN auto-detect matching an existing account) is unchanged.

## Acceptance criteria

- [x] Each queued file row in [import-select-step.component.html](../../../src/app/feature-import/components/import-select-step/import-select-step.component.html) shows a visible "+ New account" quick action alongside the existing account `<select>` — discoverable without opening the dropdown.
- [x] When `accounts()` is empty (no accounts exist in the app yet), step 1 no longer renders a dead-end disabled dropdown with zero options — it nudges the user toward the "+ New account" action (e.g. emphasized/default state and explanatory copy) for every queued file.
- [x] Triggering "+ New account" shows an inline form (name, IBAN, type) pre-filled from that file's `detectOwnIban` result when present, else from the filename with the extension stripped; `type` defaults to `'checking'`; all fields remain editable before commit.
- [x] When two or more queued files are set to create "the same" account, only one file owns the draft/creation; the other files' dropdowns list that pending draft as an option (labeled by the owning file's filename) and selecting it links the file to the same account once created — the account is never created twice for one batch.
- [x] The new account is persisted through `AccountsStore.addAccount` (→ `AccountsRepository.add`), not a direct `appDb.accounts` write, and only at commit time for the owning file (not eagerly on dropdown selection), so an abandoned wizard run creates no orphan account.
- [x] `canAdvanceFromStep1` accepts a file whose account is "pending creation" (owning or linked) as satisfied, without requiring a persisted `accountId` before step 1 is left.
- [x] Files linked to a pending draft (not the owner) resolve to the real, newly-created `Account.id` after the owning file commits, and their transactions import against that same account.
- [x] If the owning file's commit fails or the account creation errors, files linked to that same pending draft do not silently proceed against a missing account — they surface the same failure state as the owner.
- [x] Unit tests cover: draft pre-fill from detected IBAN; draft pre-fill fallback to filename when no IBAN is detected; a single owning file + ≥1 linked file resolving to one created `Account.id`; `canAdvanceFromStep1` accepting a pending draft; no duplicate account created when two files share one draft; the empty-`accounts()` nudge state rendering/defaulting to the "+ New account" action per queued file.
- [x] Verified via the fallow skill and coding-conventions skill.
- [x] Verified live in the browser, two scenarios: (1) an app with existing accounts — import two files with no matching existing account, mark the first as "+ New account", link the second to that same pending draft, commit the batch, and confirm exactly one new account exists afterward with both files' transactions attached to it; (2) an app with **zero** accounts — drop a file in and confirm step 1 visibly nudges toward account creation instead of showing a dead-end empty dropdown. Scenario (1) was fully exercised live (including the bonus opening-balance seeding). Scenario (2)'s empty-state nudge was verified via its two passing automated tests (`rowMode()` → `'empty-nudge'`, and the "No accounts yet" render) rather than live, since faithfully reaching zero accounts live would have required deleting the dev environment's real demo accounts.

## Notes

- Building on the existing IBAN-detection plumbing ([TICKET-IMP-05](./TICKET-IMP-05-fallback-iban-from-description.md) hardened the same detection area for counterparty IBANs) rather than adding new parsing logic — `detectOwnIban` already runs per file, this ticket just adds a create-account outcome for when it finds no existing-account match.
- Naming/labeling scheme for the "linked to pending draft" dropdown option is a UI-copy detail left to implementation; the requirement is that the two (or more) files are visibly and unambiguously tied together before commit.
- Out of scope: editing an already-committed pending-draft account from within the wizard after it's been created (use the normal Accounts screen/`account-form.component.ts` for that); merging a newly-created account into an existing one after the fact.
