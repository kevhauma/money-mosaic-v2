# TICKET-STAT-17 — Split trend chart into income/expense stacked-by-category bar charts

- **Area:** Statistics & Dashboard
- **Type:** Feature
- **Traceability:** extends FR-STAT-3/FR-STAT-4 (adds FR-STAT-14)

## User story

As a user reviewing my dashboard trend, I want income and expense shown as two separate, same-scale bar charts, each bar stacked by category, so I can see both how income/expense evolve over time and which categories drive each bucket's total, instead of two mirrored, category-blind bars on one shared axis.

## Description

`trend-chart-panel` currently draws income and expense as two bar series on one shared chart, with expense negated so its bars point down from a common zero line, and neither series broken down by category. This ticket replaces that with two independent charts — Income and Expense — rendered side by side within the same panel, sharing one y-axis scale so bar heights stay visually comparable between the two, with each chart's bars stacked into per-category segments (top-N categories by whole-range total, same convention `category-period-comparison.ts` already established). This single ticket supersedes the two earlier, narrower drafts (previously TICKET-STAT-17 "split into independent charts" and TICKET-STAT-18 "new category-composition panel") — this is one panel restructuring, not a plain split plus a second, separate panel.

## Desired result (to-be)

- New pure aggregate `computeCategoryCompositionTrend(transactions, categoriesById, from, to, granularity, ownSavingsIbans, accountsById)` in `core/stats/category-composition-trend.ts`, exported via `core/stats`'s barrel:
  - Selects the top-N categories (reuse a `TOP_CATEGORY_COUNT`-style constant, e.g. 5) separately for expense and income, ranked by each category's total over the **entire** `[from, to]` range (via one whole-range `computeCategoryBreakdown()` call) — not re-picked per bucket, so the same categories/colours stay stable across buckets even if their rank order shifts bucket-to-bucket.
  - Calls `computeCategoryBreakdown()` once per bucket (using `bucketKeysInRange`/`bucketDateBoundaries` for each bucket's own `[from, to]`), and for each bucket extracts just the selected top-N categories' totals (`0` for a bucket where that category has no activity) — categories outside the top-N are simply not plotted, no "Other" catch-all band.
  - Returns `{ bucketKeys: string[]; expenseSeries: CategorySeriesEntry[]; incomeSeries: CategorySeriesEntry[] }` where `CategorySeriesEntry = { categoryId: number | null; name: string; color: string; values: number[] }`.
- `trend-chart-panel` renders one card (unchanged title row: "Trend" + the existing shared `mm-granularity-picker` — bucket granularity (day/week/month/quarter) stays controllable exactly as it is today, via that one picker, and it still drives both charts at once, since they share one `computeCategoryCompositionTrend()` call) containing a `grid grid-cols-1 lg:grid-cols-2` row with two independent chart columns: Income, Expense.
- Each column is a **stacked bar chart**: one `{ type: 'bar', stack: '<income|expense>' }` series per top-N category for that column, coloured with that category's existing `category.color` (uncategorised-grey `#9ca3af` fallback for the `categoryId === null` entry when it ranks in the top-N) — same colour source `category-breakdown-panel` already uses.
- **Both charts share one y-axis scale**: compute a single shared `max` (e.g. the larger of the two columns' highest per-bucket stacked total) and apply it as both `yAxis.max`, so a bar's height is visually comparable between the Income and Expense charts rather than each auto-scaling independently.
- The dead second `yAxis` (`Net worth`) from the old combined chart is removed — it was never wired to a series.
- Clicking a stacked segment navigates to `/transactions` filtered to that bucket's date range **and** that category (`buildTransactionDrilldownParams` with `from`/`to` and `categoryId`), replacing the old bucket-only drilldown (`onChartClick` previously resolved only a bucket; it must now also resolve which category series was clicked).
- Column headers ("Income" / "Expense") are visible above each mini-chart so the split reads clearly without relying on axis colour alone.
- Responsive behaviour follows the existing breakpoint convention (`category-breakdown-panel`'s columns collapse to 1 column, stacked, below `1024px` — same rule applies here).
- If `computeTrendBuckets()` has no remaining consumer after this change, remove it and its spec (`core/stats/trend-buckets.ts` / `.spec.ts`) rather than leaving an unused aggregate in the codebase.

## Notes

- Deliberately keeps one shared granularity control rather than giving each column its own — the two charts share one `computeCategoryCompositionTrend()` call, so a second picker would either be redundant or require re-fetching the same data twice for no benefit.
- Deliberately does **not** introduce an "Other" catch-all band for categories outside the top-N — matches `category-period-comparison`'s existing top-N convention rather than inventing a new truncation rule; if that turns out to read as misleading (top-N total looking smaller than the panel's own total), a follow-up ticket can add an "Other" band.
- Net cash flow (`bucket.net`) is not displayed by either chart — it wasn't rendered by the old combined chart either. Out of scope here; a future ticket can add a net-flow view separately.
- Calling `computeCategoryBreakdown()` once per bucket (plus once for whole-range top-N selection) is the same repeated-aggregate-call pattern `category-period-comparison.ts` already uses for its window periods — accepted precedent, not a new performance concern to relitigate here.
- This ticket replaces the two previously-separate drafts (a plain income/expense split with independently-scaled axes, and a second new "category composition" panel with stacked-*area* charts) with a single merged scope: one restructured panel, one shared scale, stacked *bars* by category. No separate `category-composition-panel` component is added.
