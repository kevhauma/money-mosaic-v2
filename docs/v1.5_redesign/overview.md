# Money Mosaic — v1.5 Redesign (Overview)

[prepare.md](./prepare.md) (this version's seed notes) sets two goals: prepare the app so a new theme/style needs "as little change as possible" throughout the codebase, and then actually apply a new visual identity — Playful, Curious, Premium, Insightful, Delightful, Approachable — built around a Bento Box Grid layout, a Swiss-Modernism-2.0 type scale, Gradient-Mesh/Aurora accents, Dimensional Layering, and an OLED dark mode. Those are two different kinds of work with a hard ordering constraint between them: the redesign only stays a one-place change if every recurring visual pattern already lives in `shared/ui/` *before* the tokens change, otherwise a new color/type scale means re-touching every raw utility class a second time. This version is split into two phases for exactly that reason. Each ticketed line links to a `tickets/TICKET-*.md` file carrying its own user story, description, as-is/to-be, and acceptance criteria — this file is only the index + build order.

**New requirement family: FR-UI.** No design-system requirement family existed before this version (`grep`-confirmed across `docs/`) — FR-UI-1 through FR-UI-14 are introduced here, one per ticket below, the same way FR-INC was introduced fresh in v1.6 without a central registry edit.

**Relationship to v2's Settings track.** v2's [TICKET-SET-01](../v2/tickets/TICKET-SET-01-theme-settings.md) (light/dark/system theme switcher) and [TICKET-SET-02](../v2/tickets/TICKET-SET-02-primary-color-setting.md) (user-selectable accent palette) are not built yet and are **not** duplicated here. This version defines the actual theme *values* (a light theme and an OLED-tuned dark theme, per [TICKET-UI-11](./tickets/TICKET-UI-11-design-tokens-theme.md)) — whichever of SET-01/SET-02 lands first, before or after this version, just wires a switcher UI on top of whatever theme values currently exist in `styles.css`. Build order between the two versions doesn't matter; each makes the other's result better rather than conflicting.

**Scope decision on Flex/Grid.** Prepare.md's manual audit lists `flex` and `grid`/`grid-cols-*`/`col-span-*` among the patterns worth extracting. Generic `grid` usage still stays inline — the coding-conventions skill's styling rules already keep plain layout/positioning utilities inline in feature templates, and a generic `mm-grid` wrapping two utility classes would add indirection with no reuse value. What *does* get a primitive is the Bento Box Grid specifically — a named, reusable layout pattern the redesign calls its "layout foundation" ([TICKET-UI-03](./tickets/TICKET-UI-03-bento-grid-primitive.md)). `flex` is the one exception: 91 occurrences across 35 templates per [TICKET-UI-01](./tickets/TICKET-UI-01-styling-audit.md)'s audit, and the motivation for wrapping it isn't code reuse but readability — an editor plugin that visually hides Tailwind classes turns every `<div class="flex ...">` into an indistinguishable `<div>`, so a named `mm-flex` wrapper keeps layout intent visible even with class-hiding tooling on. [TICKET-UI-15](./tickets/TICKET-UI-15-flex-primitive.md) covers it. If a second genuinely reusable layout pattern turns up later, it earns its own primitive then; generic grid stays inline.

**Only the Dashboard gets a bespoke layout rework.** Every other routed feature (Accounts, Transactions, Categories, Import, Data) inherits the new tokens and primitives automatically through `shared/ui/` — new colors, type scale, surfaces, tables, dropdowns — without a page-specific redesign ticket. The Dashboard is singled out because prepare.md explicitly names "Dashboard-first UX" and "make the charts the hero" as design terms; every other page just needs to look consistent, not be re-laid-out.

## Considered, not ticketed yet

- **Generic grid extraction beyond the Bento pattern** — explicitly out of scope; see the scope decision above. Plain `grid`/`grid-cols-*`/`col-span-*` usage stays inline per the existing coding-conventions styling rules. (`flex` was in this category too until TICKET-UI-15 reversed it for readability reasons — see the scope decision above.)
- **Bespoke layout redesigns for Accounts/Transactions/Categories/Import/Data** — these inherit new tokens/primitives "for free" through `shared/ui/`; a page-specific layout rework for any of them is a reasonable future ticket once the Dashboard's bento treatment (UI-12) has proven out in practice, not part of this version.
- **A generic data-grid** (sorting/virtualization/column resize) as part of UI-07's table primitive — that ticket is a styling-consistency wrapper only; a real data-grid engine is a much larger feature with its own spec.
- **Expanding `LoadingSkeletonComponent` beyond list usage** (e.g. card/chart-shaped skeletons) — prepare.md flags this, but it's minor scope; fold it into whichever Phase A ticket happens to touch the same panel, or file it separately once UI-01's audit shows how many call sites would actually benefit.
- **New icon set or illustration system** — out of scope; `@ng-icons/tabler-icons` stays as-is, this version is about layout/color/type/surface, not iconography.
- **User-facing theme switcher / accent-color picker** — that's v2's [TICKET-SET-01](../v2/tickets/TICKET-SET-01-theme-settings.md)/[TICKET-SET-02](../v2/tickets/TICKET-SET-02-primary-color-setting.md), not duplicated here; see the relationship note above.
- **New animation dependency** (`@angular/animations` or a third-party motion library) — [TICKET-UI-13](./tickets/TICKET-UI-13-chart-visual-language.md) covers "Motion-Driven" using ECharts' own animation config and CSS transitions; no new package is justified for this version's scope.

## Definition of Done (applies to every ticket)

Per [../../CLAUDE.md](../../CLAUDE.md): `ng lint` + `ng test` + `ng build --configuration development` all pass, plus the fallow skill, plus a live browser check for any UI-visible change (ask the user first per this repo's verification rule, and continue without it if declined). **No Dexie schema changes in this version** — every ticket is styling/markup/token-only; current schema stays at `.version(11)`. **The production bundle budget in `angular.json` is never raised** — Phase B's motion/visual work stays inside ECharts' existing animation engine and CSS, no new dependency. Every new `shared/ui/` primitive follows the existing conventions: `mm-`-prefixed selector, typed string-union `input()`s over raw daisyUI classes, `class = input('', { alias: 'class' })` passthrough, `ControlValueAccessor` for any form-control wrapper, its own component folder with a colocated `.spec.ts`. Components/stores continue to never touch `appDb` tables directly — not applicable to most of this version's tickets, but still holds since none of them introduce new data access. Rules must still never overwrite a manually-set category (`categoryManual`) — unaffected by this version, called out here only because it's a standing hard rule, not because any ticket touches it.
